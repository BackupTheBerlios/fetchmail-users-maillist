After investigation with data that was sent in private it turns out this  
is a SSL misconfiguration of the server, apparently as limitation in a  
basic unix-based shared web hosting plan of AT&T (and thus likely an  
incentive that users book a more expensive plan).

Fetchmail versions 6.3.9 and newer have a "--sslcommonname" option that  
can be used to override the common name fetchmail will expect in the  
certificate, like adding this to your command line:

	--sslcommonname "*.sslcert35.com"

Or without the leading hyphens (dashes, --) to your .fetchmailrc file.

Depending on the upstream certification policies, it may also be advisable  
to use --sslfingerprint + fingerprint (please see fetchmail's manual page)  
as well, providing that the provider offers the fingerprint (MD5 in  
fetchmail's case) through a separate secure channel (for instance, via  
phone line - not VoIP! - faxback, printed on their paperworks, or similar)

The proper solution though is to have the upstream server use proper  
certificates, if this is affordable.
