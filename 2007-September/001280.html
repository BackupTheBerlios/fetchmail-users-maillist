It is designed to support that, yes.

> Is that right?
> If so, can you please tell me the correct method for starting it up after a server restart?

There are too many options to call one of them canonical. ;-)

> I understand from everything that I have seen, that the SuSe standard init script is bad,
> but I don't know what I am supposed to use.

First check if you use the mda option with an MDA (usually procmail with
SUSE) that was *not* installed setuid-root. Either use SMTP in that case
or set the setuid flag if the MDA is designed to support that.

Ideas, untested, but should work:

- modify the init script to run fetchmail like
  "su nobody -c fetchmail --options"

- @reboot lines in users' crontabs (if your cron daemon is new enough)

- write your own init script that drops privileges

- write an inittab line that drops privileges (has the additional
  advantage of restarting fetchmail) - for instance "su nobody -c
  fetchmail --nodetach"

- use "runit" <http://smarden.org/runit/> to start fetchmail with su and
  nodetach (see above)

- switch to FreeBSD where the fetchmail port from a current ports tree
  has a decent rc script and a clueful package maintainer (that's not to
  say Stanislav was clueless or Petr Cerny (the new maintainer for
  openSUSE's fetchmail package) were, not at all; they just had internal
  regression test failures that prevented them from quickly changing the
  script for SUSE <= 10.2. I haven't yet seen 10.3.)
