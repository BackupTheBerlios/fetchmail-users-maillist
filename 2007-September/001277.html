SUSE run fetchmail as root, which is discouraged, so a bug report was
filed. As long as you inject into SMTP and/or maildrop or procmail are
installed setuid-root (both are designed for that purpose), you can run
fetchmail as "nobody" or some other unprivileged account.

While I'm not aware of code injection vulnerabilities in fetchmail, I
may have overlooked a bug and thus it's simply due diligence to not hand
fetchmail root privileges when it doesn't need them.

I considered automatically dropping privileges, but didn't want to make
this incompatible change in 6.3.X.
