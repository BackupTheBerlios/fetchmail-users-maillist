Yes, and if you fetchmail <mailserver> it will (if i recall)
fetch all three accounts at once, which is undesired
if you wish to keep them separate as to say which
VPN they go over in time, etc.
Luckily 'via' worked to override the (sortof like 'via') 'poll'
keystring.
It's a hack, will likely move to an rc file per account
or fdm or something.

Since it's unlikely poll will ever be turned into a simple
fetchmailable rc account label that has no underlying
split meaning as to 'server/hostname/tcp', and making
via required, I guess we need such a label in the next
release: fetchmail -A <accountlabel>
In fact, keeping poll as the (sortof) grouping option
it is today could be useful.
But I don't like the split meaning of it.

To fetchmail's credit, the behavior of poll/via is documented
as such in the man page under:
 Keywords Not Corresponding To Option Switches

Though I would definitely remove/rewrite this part:
 The 'via' option is for if you want to have more
 than one configuration pointing at the same site.

>        user "user2 at mydomain.de"
> poll mailserver.myprovider.de with proto POP3 interval 3
>        user "user3 at mydomain.de"
