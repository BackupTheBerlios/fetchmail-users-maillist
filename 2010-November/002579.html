     You are using fetchmail's -d option, or "daemon mode" option, and 
you are also using the "fetchall" keyword, according to the .fetchmailrc 
you posted.

     You can use those two options together, but not if "keep" has been 
specified.  This post explains why:

http://oclug.on.ca/archives/oclug/2002-January/014584.html


     But why is fetchmail acting like "keep" is in your fetchmailrc (or 
-k on the command line)?  I don't see -k below or the word keep in your 
fetchmailrc.

     Maybe there is an old fetchmailrc daemon running  off a different 
.fetchmailrc?  (First, kill all fetchmail processes and retry using the 
.fetchmailrc you posted.)

     Next, explicity add the "nokeep" option (-K) and see what happens.  
(Is it possible the "expunge 50" is triggering a bug in fetchmail?  It's 
the only option that seems related to the deletion of emails, like "keep".)

     Next, how did you install and configure fetchmail?  You mentioned 
Debian.  If you installed the .deb package, then there are files in 
places like /etc/default/fetchmail that you should look at.

     Next, you can use strace (as in, "strace -p $PID_OF_FETCHMAIL") to 
see what files are being opened and read in at launch time.

     Next, turn on full logging of the fetchmail and post.  Perhaps some 
combination of "-d" with "expunge 50", plus ETRN or ODMR, is causing 
this symptom.

     Please let us know how the testing goes.


Thanks,
