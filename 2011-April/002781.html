Well, if you mean "safe" as in "reliable", then yes, in this particular
case (no weird library dependencies, no major changes in the Debian
packaging that could affect the functionality) my backported package
ought to be as reliable as the fetchmail-6.3.19-1 version in Debian
testing.  If you mean "safe" as in "secure", well then, mostly yes - in
this particular case my backported package ought to be as secure as the
6.3.19-1 version in Debian testing right now, and also as secure as the
"real" fetchmail-6.3.19 distributed by Matthias Andree :)  You can
always check the sources if you're feeling particularly paranoid :)

> Here is the result from uname -a :
>  
> Linux mail 2.6.26-2-686 #1 SMP Thu Nov 25 01:53:57 UTC 2010 i686 GNU/Linux

Great, this means you're on an i386 system, same as me, so you can use
directly the binary packages I've just uploaded to
http://devel.ringlet.net/mail/fetchmail/debian/ - download the .deb
files and install them using "dpkg -i".

You may verify the integrity of the download by installing the
"devscripts" package and running "dscverify" on the *.changes file; it
will most probably complain that the PGP key that signed the *.changes
file is not in APT's keyring - this may be overcome by downloading my
PGP key from the URL in my signature and passing it to "apt-key add".
Note that this will mean that your APT will trust all packages and
repositories signed by me :)  It's your call as to how much of a problem
that is (and since I'm already in the Debian Maintainers keyring and
it'll probably not take much longer than a couple of months more for me
to complete my full Debian Developer application process, I dare say
that it might not be too much of a problem, even if I do say so myself
:P )
