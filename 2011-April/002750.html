Am 21.04.2011 11:57, schrieb Martin Koeppe:

> I don't think of oversized messages due to some limit setting (I don't
> use a limit setting), but a smtp error from the receiving server that
> the mail is too big. In this case, it shouldn't (for me) be bounced and
> deleted, but instead left alone on the POP3 server. I think this is what
> softbounce is useful for.

Limits are deterministic or temporary (due to "out of disk space").
Either case is handled properly.

> So, apparently, (1) antispam and softbounce are (currently) not useable
> together, and (2) the limit setting is probably handled differently from
> the error condition that the smtp server rejects a message due to a size
> limit.  (2) might be reasonable because when using limit there is no
> transfer of the message to fetchmail.  But (1) seems misdesigned
> somehow.  Also: if softbounce overrides antispam, then it's confusing
> that the SMTP error (invalid From: address in my case) is only logged if
> antispam is not enabled. So antispam always influences log output, but
> is honored only if no softbounce.

Just set the fetchmail message size limit to the maximum that your
destination SMTP or LMTP server can handle, and you'll never bump into a
condition where fetchmail pulls a message too large for the SMTP behind
to handle. Leave 1 kB of room for added headers.
