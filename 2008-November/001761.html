> Reason for "why not the sender" is that the recipient address (in
> singledrop) is trustworthy, whereas the sender address might be forged. If
> you'd blindly notify senders, then you might end up hitting innocent
> bystanders with such notifications. Usually we should avoid that. See
> "Backscatter" in the Postfix documentation (there's a dedicated README file).
>   
Right, it is a problem, agreed.
> Bouncing after the fact is a bad idea anyhow - actually anything behind the
> MX should not be more restrictive than the MX itself, in order to let the
> MX decide and reject undesired messages (too big, wrong sender, wrong
> content) during the original SMTP session.
>
>   
I know, but there are wide open providers in the world, still believing 
internet is a peaceful place, accepting anything which ist dropped to 
the users mailbox. In some constellations, especially with very small 
customers with dynamic internet connections, it's not possible to take 
over the MX, then we're using fetchmail to get the mails to the users, 
but often we're more strict than the providers. So what we're doing is 
getting the mail, filter for spam, viruses, dangerous attachements and 
message size. But we're the users friends and want wo protect them, even 
if they not always agree with us (especially message size limit).
