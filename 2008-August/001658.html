Quoting from Christian Ebert's mail on Wed, Aug 06, 2008:
> machine -> Speedport w701v (router) -> Speedport 300HS (vdsl modem) -> splitter

Try disabling/changing the escape character of the vdsl modem then.
Note that this issue can occur if there is a modem anywhere in the
link between you and your ISP. Could you also report the existing
escape character of that modem?

> > You might be hitting the issue
> > briefly mentioned in FAQ X7. If your router has an escape character of
> > "+" and a mail contains "+ + +" (*), the router may intercept the text
> > as a modem escape sequence. Your options in that case are:
> > 
> > a) Disable the modem escape character, or
> > b) Change the modem escape character to some non-ascii character which
> > does not occur in mails, or
> > c) Download using ssl! (**)
> 
> I poll most of my accounts using ssl, doesn't work either.
> 
> Seems to be a prob with fetchmail in the end -- see my other mail
> wrt mpop -- or fetchmail is stricter in some standards or has a
> different way to do the server handshake or ...

Note that modems *normally* interpret the escape sequence only if it
is followed by a pause. So, the exact sequence is "+ + + <pause>". It
is quite possible that this sequence is occuring at the same location,
possibly at a packet boundary, everytime with imap. And, at a
different location with pop3.

Another explanation: If your modem escape character is "*" and if some
imap response has "* * *", you have trouble. pop3 responses will
probably be different and not cause the problem.

Note that you have mentioned problems with fetchnews also, so it is
unlikely that the problem is with fetchmail's implementation.
