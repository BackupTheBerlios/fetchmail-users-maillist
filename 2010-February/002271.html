>> The most-requested features that I will never add, and why not:
>>
>> Password encryption in .fetchmailrc
>>
>> The reason there's no facility to store passwords encrypted in the
>> .fetchmailrc file is because this doesn't actually add protection.
>>
>> Anyone who's acquired the 0600 permissions needed to read your
>> .fetchmailrc file will be able to run fetchmail as you anyway -- and if
>> it's your password they're after, they'd be able to rip the necessary
>> decoder out of the fetchmail code itself to get it.
>>
>> All .fetchmailrc encryption would do is give a false sense of security

> ... I don't agree with you. Example: while server administrator is
> editing fetchmailrc config file, a colleague walk around him and can see
> all password in clear text.

Press Windows+L, Ctrl-Left + Ctrl-Right + L or similar to blank out and  
lock the screen, or similar combinations depending on your desktop and  
window managers.

You can keep passwords in a separate .netrc file, separate from  
.fetchmailrc.

You can also use client-certificate-based authentication.

There are protector foils for screens (not sure up to which size, surely  
for laptops) that prevent spying on the screen from the side, i. e. you  
can only see from the center.

> I can think a lot of example like that.
>
> I agree with you that crypt password isn't an hard security solution,
> but it's better than nothing. I think.

It requires special support for editing the configuration files from  
inside fetchmail. Particularly, fetchmail will have to be able to re-write  
the configuration file, and somehow transport unknown options and comments  
to the output -- this is a major undertaking. There are more important  
code changes up the priority lists.

Note that crypt() isn't suitable because it's a one-way function, but you  
need to be able to recover the original password from the encrypted output.

> Can you suggest me files source used to read fetchmailrc?

*.[chly] - try fetchmail.*, rcfile* first, or possibly use "cscope" or  
other static/source call analyzers.

Note that relevant patches will probably not be accepted into the  
fetchmail source.
Anything that creates a false sense of security will be rejected.

If you think a change improves security, I will ask for documentation on  
the exact attack scenario that is being addressed (you provided it here,  
but based on that, *I* would not change fetchmail, there are features  
missing that are more important).

Sorry.
