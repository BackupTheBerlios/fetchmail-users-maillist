Good, but it looks like a qmail problem.

> 2008-05-30 09:14:27.906562500 fetchmail: SMTP> DATA
> 2008-05-30 09:14:27.925013500 fetchmail: SMTP< 354 go ahead
> 2008-05-30 09:14:27.927412500 
> #*********************************************.**************************************.****************$
> 2008-05-30 09:19:29.636029500 fetchmail: smtp listener protocol error
> 2008-05-30 09:19:29.636049500  not flushed
> 
> qmail_smtp:
> 
> 2008-05-30 09:14:27.855293500 tcpserver: status: 1/20
> 2008-05-30 09:14:27.886063500 tcpserver: pid 9406 from 127.0.0.1
> 2008-05-30 09:14:27.886082500 tcpserver: ok 9406 
> mail.myserver.co.uk:127.0.0.1:25 localhost:127.0.0.1::33893
> 2008-05-30 09:19:30.815287500 tcpserver: end 9406 status 256

This is not success - it means whatever process was spawned (likely
qmail-smtp) exited with a status code of 1 (on many systems, the lower 7
bits are the signal, bit 7 signals a core dump, and bits 8-15 are the exit
status), and apparently the SMTP process exited prematurely.

What kind of POP3 server is the upstream site using? Is it doing proper
byte-stuffing and is its LIST command using proper sizes? qmail toasters
and some Exchange configurations are known to provide bad SIZE numbers.
