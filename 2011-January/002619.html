Hi Sunil,

On Sun, 23 Jan 2011, Sunil Shetye wrote:

> Hi Martin,
>
> --- On Sat, 22/1/11, Martin Koeppe <mkoeppe at gmx.de> wrote:
>
>> The man page says:
>>> Fetchmail's method of sending bounces due to errors or
>> spam-blocking and spam bounces requires that port 25 of
>> localhost be available for sending mail via SMTP.
>>
>> I think the 2 points above should be made more explicit in
>> the man pages, e.g. when describing "set (no) bounce",
>> "smtphost", "esmtpname" and/or "esmtppassword".
>
> Please note that the 'smtphost' and other options are related to 
> acceptance of local mails. This means that 'smtphost' is only 
> guaranteed to accept mails for local users and for the postmaster.
>
> It cannot be assumed that 'smtphost' is guaranteed to relay the 
> bounce mails even when 'esmtpname' and 'esmtppassword' are 
> specified. For all you know, the 'smtphost' might not even be 
> connected to the internet even indirectly.

Yes, I saw this problem already with cygwin's 6.3.9. But 
fortunately we now have 6.3.19 in cygwin with support for soft 
bounces, so for me the use of postmaster delivery isn't necessary any 
more.

Because in this case I use cygwin and have no "normal, unix-like" 
local mail system, but only Exchange which required in default 
configuration esmtpuname/esmtppassword. I used as smtphost 
"127.0.0.1", which works, but as you write for postmaster access the 
fixed server "localhost" is used, on Windows 2008R2 this gets resolved 
to ipv6 ::1 instead of 127.0.0.1, and Exchange has 2 distinct policies 
for ip4 and ip6. At least I now know that smtphost isn't always 
honored, and why mails to postmaster weren't delivered while other 
recipients worked ok.

When simply reading the man page entry for smtphost, I didn't thought 
automatically that far that (1) relaying of bounces might be a problem 
AND (2) that fetchmail already tries to workaround this by ignoring 
smtphost for bounces. So I would find it helpful to add these 2 
points to smtphost description. And also that esmtpname/esmtppassword 
are only applied to smtphost, and not the fixed "localhost" for 
bounces.

> Yes, fetchmail's bounce behaviour requires more work. However, that 
> is not going to be trivial as any implicit assumptions are only 
> going to muddle the issue further.

For my purposes, the soft bounces (i.e. letting the mail where it is 
when it can't be forwarded) are currently perfectly ok.

Martin
