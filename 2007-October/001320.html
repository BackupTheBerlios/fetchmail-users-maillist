On 10/21/07, Joe Acquisto <joea at j4computers.com> wrote:
> Sigh.  As you may know, this if version 6.2.5.  Mulitdrop.  Works well enough, but BCC's are not working.

This suggests that either your ISP isn't using envelope-to type
headers, or your configuration isn't correct.

> The ISP is using modus, vircomm, or some such.
>
> The problem is that a single message is being delivered, only for the direct addressee's.  A Bcc message is never seen, in the mailbox.  I've looked at the headers.  (I disable fetchmail and to a webmail thing into the mailbox). Of course, fetchmail can know nothing about a Bcc in that case.
>
> The ISP acknowledges this and has attempted to work around this by creating individual mailboxes for each user and "forwarding" their mail to a single drop.  That does not seem to work well.  Fetchmail now rejects mail with a 550 error, no such recipient.  I don't know why, but the header may be the problem.

More details on that may help get that working.  A 550 error would be
from your local SMTP server and suggest that you had an error in your
own setup somewhere.

> A typical header is below as is my .fetchmailrc.
>
> ****************************************************
> Resent-To: allmail at x.com

That's a potential candidate for the envelope header.

> ************************************************************
> # Configuration created Fri Oct 19 14:14:53 2007 by fetchmailconf
> set logfile "/var/log/fetchmail.log"
> set postmaster "admin at daybreakoffice.com"
> set bouncemail
> set no spambounce
> set properties ""
> set daemon 120
> #poll nn.nnn.nnn.nnn via mail.a.net
> # with proto POP3 envelope Delivered-to
> #    localdomains x.com
> #     user 'allmail at x.com' there with password 'foo' is * here
> #    smtphost  192.168.0.xxx
> poll nn.nnn.nnn.nnn via mail.a.net
>  with proto POP3 envelope Received

Received header parsing is known to be problematic and will (from
memory) be removed from a future release of fetchmail.

>     localdomains x.com
>      user 'allmail at x.com' there with password 'daybreak' is * here
>     smtphost  192.168.0.xxx
>
> *********************************
>
> In the uncommented lines, I attemped to change the "envelope" but apparently am off base.

Try the "Resent-To" header.  However I think your only choices are to
move away from multidrop or change ISP.
