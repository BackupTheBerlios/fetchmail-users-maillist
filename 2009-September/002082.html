> On Fri, 11 Sep 2009 15:52:23 +0200
> Matthias Andree <matthias.andree at gmx.de> wrote:
>
>> Gerard schrieb:
>>
>> > I am not sure about your "outbound mail statement". I was not
>> > referring to outbound mail. I was simply pointing out what I
>> > thought would be a useful addition to fetchmail. IMHO, it could
>> > make seamless integration to systems that employ security
>> > techniques easier. I have several utility programs; i.e.
>> > "DenyHosts", etc. that support this feature.
>>
>> The default destination host of "localhost" is secure by definition,
>> and having fetchmail inject the fetched mail over insecure/untrusted
>> links doesn't fit my view of the use scenario.  I'd appreciate some
>> elucidation on the use or attack scenario.
>
> Basically, the system, which is used by several different users, was set
> up to require SASL authentication and encryption to make the system
> secure. Without some form of security, anyone who could connect to the
> local network could send mail through it. By requiring SASL
> authentication, I have reduced that possibility. In order to get
> fetchmail to work, I removed the requirement for the use of encryption
> and now am allowing anyone with local access to access the mail server.

You can require authentication for outbound email only, and give up on it  
for inbound mail; at least Exim and Postfix should allow such  
configurations.

> I should probably add that the Postfix mail server also accepts mail
> from the Internet. It is not strictly being used as a back-end for
> fetchmail.

You can easily tell Internet from local use by referring your  
configuration options such as smtpd_recipient_restrictions to mynetworks  
(putting permit_mynetworks before more restrictive options that require  
authentication). I am aware that this is currently less convenient than if  
fetchmail supported SASL, but I don't see security issues here unless you  
let untrusted user log into the computer that runs Postfix/Fetchmail.

The key point however is that Postfix policies that depend on recipient  
address or client IP are available now, whereas fetchmail support for  
SMTP/TLS won't be happening on short notice. Particularly not in release  
quality packages. I'm afraid that (a) I have a full-time job that isn't  
about fetchmail, (b) apart from MAPI, I'm apparently the only contributor  
for anything nontrivial.
